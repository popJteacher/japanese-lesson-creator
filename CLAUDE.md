# CLAUDE.md — プロジェクト統一プロトコル

> このファイルには「変わらないルール」だけを置く。
> 現在地・やることは書かない（→ `NEXT_ACTIONS.md`）。歴史も書かない（→ git / `handoff_archive.md`）。
> このファイルが長くなり始めたら、それは「何かを別ファイルに分けるべき」サイン。

---

## このプロジェクトは何か

教師が文型データをアップロードすると、スライド／宿題／アクティビティ HTML・教案 docx が
自動生成される教師向け Web ツール（`japanese-lesson-creator`）。
語彙・例文の画像・音声は GAS パイプラインで生成し Google Drive に蓄積、
レジストリ経由で教材に接続する。将来マネタイズ予定。

---

## 最重要ルール：状態はどこにあるか

このプロジェクトは過去、引き継ぎ文書が肥大化し（917 行・追記のみ運用）、
チャット間で情報が繰り返し失われた。原因は **「永続アーカイブ」と「今やることの briefing」を
同じ文書に兼任させたこと**。二度とやらない。

| 種類 | 置き場所 | ルール |
|---|---|---|
| 今の状態・次の行動 | `NEXT_ACTIONS.md` | 1 ページ以内。毎回まるごと書き直す。解決済みは削除する。**追記のみ禁止（むしろ逆）** |
| 過去の決定・経緯・バグ史 | git 履歴 ＋ `handoff_archive.md` | 新セッションは原則読まない。特定の過去判断を調べる時だけ参照 |
| 不変の仕様・スキーマ | `docs/REFERENCE.md` | 参照用。安定。プロトコルではない |
| 各ステップの分類・分業 | `PIPELINE_INVENTORY.md` | Claude Code が実ファイルから生成・維持する。語りではない |
| 移行ロードマップ | `docs/MIGRATION_PLAN.md` | Phase 0〜4 の順序・各 Phase 完了条件 1 行。NEXT_ACTIONS は active な Phase のみ |
| main / worktree の役割分担 | `docs/WORKFLOW.md` | ファイル所有権 / 同期方向 / merge ポリシー。1 セッション = 1 worktree |

**状態はナレーションではなく機械導出する。**
新しいセッションは前任者の文章を信じない。最初に検証コマンドを走らせ、
ファイルから真実を再導出する。`NEXT_ACTIONS.md` の記述とコマンド出力が
食い違ったら、**コマンドが正**。

## チャットと Claude Code の分担

**Claude Code に任せる（＝原則すべて）：**
- `.gs` ソースの著作・編集・バージョン管理（実行はしない＝人間が GAS に貼る）
- プロンプト JSON・Python 後処理・lesson JSON の著作と検証
- バリデーター／レポート／ビルド等のスクリプト
- 決定論的なデータ処理（語彙抽出・分類ロジック・整形）

**チャットに残す（＝この残余だけ。小さく保つ）：**
- 新課の例文・教育設計の対話的レビュー（人間が一文ずつ自然さ・教科書整合を判断する部分）
- 生きた GAS の実行・スプレッドシート／Drive の目視確認（Claude Code は実行不可）

チャット面積が小さいほど引き継ぎ損失が減る。迷ったら Claude Code 側に寄せる。

## Claude Code 自律ルール

**自分で進めてよい：** 原因が明確なバグ修正／動作を変えないリファクタ／
検証・レポート生成／既知リストとの照合／`NEXT_ACTIONS.md` の更新。

**人間に報告してから：** 設計方針の変更／スコープ追加／
トレードオフのある複数解／教育内容（例文・語彙・文型）の確定。

## セッションの開始と終了

> どのチャネルで何を頼むかは `docs/SESSION_START.md`（1 ページ）。

**開始（毎回この順）：**
```
1. CLAUDE.md を読む（このファイル）
2. 検証コマンドを走らせて現状を機械導出する
   - npm run validate        （lesson_NN.json 検証 ※未実装ならステップ2で作る）
   - npm run missing-assets  （imageUrl/audioUrl が null の一覧 ※同上）
3. NEXT_ACTIONS.md を読む（コマンド出力と矛盾したらコマンドが正）
4. 作業する
```

**終了（毎回）：**
```
1. NEXT_ACTIONS.md を「今の真実」でまるごと書き直す（解決済みは消す・1ページ厳守）
2. 重要な過去判断が生じたら handoff_archive.md に1行追記（NEXT_ACTIONS には残さない）
3. git commit（履歴はここに残る。文書に履歴を書かない）
```

## 旧資料の扱い

`handoff_v4_4.md`・`project_handoff_complete.md`・旧 `CLAUDE.md`・各 `*_handoff_*.md` は
**アーカイブ扱い**。`handoff_archive.md` に集約し、新セッションは原則読まない。
これらの本文（特に状態・進捗の記述）は **再導出されるまで信用しない**。

ただし中の仕様・スキーマ・バグ対処・確定設定値は価値があるので、
**削除ではなく `docs/REFERENCE.md` / `handoff_archive.md` へ移設して保全する（消さない）**。
情報は消さない・置き場所を変えるだけ。

## 禁止事項

- 引き継ぎ文書を「追記のみ・削除禁止」で運用すること（`NEXT_ACTIONS.md` は逆）
- 状態を文章で語って次セッションに信じさせること（必ずコマンドで再導出させる）
- ライン A／ライン B を別リポジトリ・別管理にすること（制御面は 1 つ）
- このファイルに現在地・履歴・スキーマ詳細を書き込むこと
