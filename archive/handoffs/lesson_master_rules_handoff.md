## 2. 前提：プロジェクト概要

### 作るもの
教師向けHTML自動生成Webツール。教師がファイル（文型データ）をアップロードすると、以下の4つの出力物が自動生成される。

| 出力物 | 用途 |
|---|---|
| スライドHTML | 授業中に画面に映す |
| 宿題HTML | 授業後に学習者が使う |
| アクティビティHTML | 活動中に学習者が使うタスクシート |
| 教案docx | 教師が授業準備に使う |

### データの3層構造
```
lesson_NN.json（課マスター）
  └── 「この課で何を教えるか」文型・語彙・例文・フロー

session_NNN.json（セッション）
  └── 「この授業でどの文型をどの順で教えるか」

activity_catalog.json（活動カタログ）
  └── 全課共通の活動定義（現在18種類）
```

### 使用教科書
**「日本語の教え方ABC」を唯一の基準教科書として採用。**

理由：みんなの日本語・げんきが扱う文型をすべて網羅しているため。
判断：将来「みんなの日本語」「げんき」に完全移行する場合はシステムを使う意味がなくなるため、ABCベースで固定する。他教科書への移行が必要になった場合は `lesson_NN.json` の中身とT/Lスクリプトを差し替えるだけで対応可能（エンジン側は変えない）。

## 3. 確定済み：課マスター作成ルール全14項目

### A：必須ルール（これがないと課マスターが壊れる）

#### A-1　例文はPDF照合してから書く
- 教科書PDFをその課の分だけ読み通してから例文を書く
- 「先に草案を書いて後で照合」は禁止
- 照合済みであることを `originalSources` に記録する

#### A-2　登場人物を勝手に作らない
- 教科書に登場する固有名詞（リンさん・アインシュタイン等）はそのまま使う
- それ以外の人物が必要な場合は `Aさん/Bさん/Cさん` 等の教科書記号を使う
- 独自キャラクター（田中先生・山田さん等）は作らない

#### A-3　教科書の固有名詞は変えない
- 教科書に書いてある固有名詞はそのまま使う
- 変えた場合は必ず `originalSources.replacementNote` に理由を記録する

#### A-4　canDoは「〜できる」の形で1文型につき1文
- 形式：「〜できる」で統一
- 数：1文型につき1文のみ
- 良い例：「『これ・それ・あれ』で近くのものと遠くのものを指し示せる」
- 悪い例：「指示詞を使う。これ・それ・あれの違いがわかる」（2文・形式バラバラ）

#### A-5　語彙導入の教具を必ず記録する
語彙リストの各語に `vocabType` を付与する。選択肢は以下の4種類：

| vocabType | 説明 | 代表課 |
|---|---|---|
| `実物` | 教師が実際のものを持ってくる | 第2課（時計・ペン等） |
| `語彙カード` | 1語1枚の絵カード | 第1課（人物・職業） |
| `場面絵` | 複数要素が入った絵・地図 | 第7課（公園マップ） |
| `対比絵` | 2つを並べた対比の絵 | 第6課（大小かばん） |

#### A-6　画像の役割（imageRole）を例文・語彙ごとに明記する
`imageRole` フィールドを必須とする。選択肢：

| imageRole | 説明 |
|---|---|
| `vocab_person` | 役割不明の人物絵カード |
| `vocab_object` | もの絵カード |
| `scene` | 場面絵（複数要素を含む） |
| `contrast` | 対比絵（2つを並べる） |

背景：「医者の格好をした人の画像が表示されるが、それが誰かわからない」という問題を防ぐため。

---

### B：品質ルール（これがないと課によってバラつく）

#### B-5　語彙の記録のしかた（3点セット）
① JLPTレベル（N5〜N1）を書く
② 「この課が初出か・前の課で習ったか」を書く（`_inheritedFromLessonX`）
③ 絵カード化できない語（「わたし」「です」等）は理由を `_note` に書く

#### B-6　practiceTemplatesは最低2つ
- 1文型につき必ず2つ書く
- 標準構成：「肯定文テンプレート」＋「疑問文または否定文テンプレート」
- 形式：`＿＿＿` 形式の穴埋めで統一

#### B-7　materialNeeds[]の付与ルール
- `intro_activity` と `main_activity` には**必須**
- `pattern`・`example` はオプション
- `wrapUp`・`review`・`intro_slide` には付与しない（schema規則）

#### B-8　推奨時間の目安

| ステップの種類 | 目安の範囲 |
|---|---|
| 復習 | 3〜5分 |
| 導入アクティビティ | 5〜10分 |
| 文型1つ | 3〜7分 |
| メインアクティビティ | 10〜20分 |
| まとめ | 2〜5分 |

目安であり、実際の時間は `session_NNN.json` で上書き可能。

#### B-9　活動例の必須フィールド
ABCの活動例を課マスターに取り込むとき、以下を必ず書く：

| フィールド | 内容 |
|---|---|
| 活動名 | 例：「忘れ物探し」 |
| 設定 | どんな場面か（1〜2文） |
| 準備物 | 何を何枚用意するか |
| 会話例 | 最低1往復 |
| fadingStage | 5段階のどこか（stage1〜5） |
| ABCとの対応 | activity_catalog.jsonのどのIDと対応するか |

---

### C：運用ルール（作業効率に関わる）

#### C-9　作業ステップの順序
この順番を必ず守る：
1. **Step 1a**：骨格（文型の一覧と構造のみ）
2. **Step 1b**：語彙リスト（PDFを読んでから）
3. **Step 1c**：例文（PDFと照合しながら）
4. **Step 1d**：materialNeeds（例文が確定してから）
5. **Step 1e**：アクティビティ選定（ABCの活動例とcatalogを照合）

例文より先にmaterialNeedsを書かない。アクティビティ選定は最後。

#### C-10　変更履歴の書き方
`_meta.changes` には「変更内容」と「変更理由」の両方を必ず書く。
理由のないエントリは不完全とみなす。

#### C-11　バージョン管理の分離
- `formatVersion`：全課共通のスキーマ版数（現在 `2.7` が最新）
- `lessonVersion`：この課だけの改訂番号（`1.0` から始める）
- 両方を `_meta` に並べて書く

#### C-12　教案の手順はABC準拠で書く
- T/Lスクリプトを含む詳細な手順を教案docxに出力する
- 課マスターには `teachingHint`（1〜2文の要点）だけ書く
- 教案には「日本語の教え方ABC 第N課 教え方の例に準拠」と明記する
- 理由：ABCの教え方を忠実に実装することがこのシステムの最大の差別化要因であるため

### 振り分けの3原則
1. **スライドと宿題は学習者向け、教案は教師向け**（カテゴリ4・6・7は教案のみ、例外なし）
2. **アクティビティは「学習者が活動中に参照するもの」に限定**（活動設定・会話例・語彙カードのみ）
3. **プラスαは全出力先で補足扱い**（必須情報と混在させない）

## 5. 確定済み：システム・課マスターの問題点と改善方針

### レベル1：第3課を作る前に解決すべき問題（優先度：高）

#### 問題1：`activity`セクションの廃止
lesson_01.jsonの末尾にある `"activity": { "selectedId": null }` は旧設計の遺産。現在は `flow[].activityId` で管理する設計に移行済みなのに残っている。

**対応：** lesson_01・02から削除。第3課以降では作らない。

#### 問題2：フィールド名の統一
ABCには「注意」と「プラスα」という明確なラベルがあるが、課マスターでは `note`・`plusAlpha`・`_comment` が混在している。

**対応：** ABCの「注意」→ `cautionNote`、「プラスα」→ `plusAlpha` に統一。`note` は廃止。

#### 問題3：語彙情報の二重管理
`patterns[].vocabulary[]` と `flow[].materialNeeds[]` に同じ語彙情報が重複して書かれている。

**対応：** 語彙情報は `patterns[]` のみで管理。`flow[].materialNeeds[]` には「教具の種類」だけを書き、詳細は `patterns[]` を参照する設計にする。

#### 問題4：ABCの活動例とactivity_catalogの対応が未記録
ABCの活動例（忘れ物探し・落としもの返しごっこ等）と `activity_catalog.json` の18種類のアクティビティとの対応関係がどこにも書いていない。

**対応：** 課マスター作成時に `ABCactivityRef` フィールドを追加してABC活動例名とcatalog IDの対応を記録する（C-9のStep 1eで実施）。
